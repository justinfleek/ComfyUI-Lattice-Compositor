# BUG-183: NaN Bypasses Normalization in expressionEaseIn()

## Summary
Same vulnerability as BUG-182 but in expressionEaseIn(). Division by (tMax - tMin) without validation.

## Severity: MEDIUM

## Location
- File: ui/src/services/expressions/expressionEvaluator.ts
- Lines: 197-214
- Function: expressionEaseIn()

## The Bug
```typescript
export function expressionEaseIn(
  t: number,
  tMin: number,
  tMax: number,
  vMin: number | number[],
  vMax: number | number[]
): number | number[] {
  let normalized = (t - tMin) / (tMax - tMin);  // div/0 not protected!
  normalized = Math.max(0, Math.min(1, normalized));  // NaN bypasses!
  // ...
}
```

**Identical crash paths to BUG-182:**
- `tMin = tMax` → division by zero → NaN
- Any NaN input → NaN propagation

## Code Evidence
```typescript
// Lines 197-214
export function expressionEaseIn(
  t: number,
  tMin: number,
  tMax: number,
  vMin: number | number[],
  vMax: number | number[]
): number | number[] {
  let normalized = (t - tMin) / (tMax - tMin);
  normalized = Math.max(0, Math.min(1, normalized));

  const eased = normalized * normalized * normalized;

  if (Array.isArray(vMin) && Array.isArray(vMax)) {
    return vMin.map((v, i) => v + (vMax[i] - v) * eased);
  }

  return (vMin as number) + ((vMax as number) - (vMin as number)) * eased;
}
```

## Impact
Same as BUG-182 - NaN corrupts eased animation values.

## Test Cases
```javascript
expressionEaseIn(5, 5, 5, 0, 100) // Returns NaN
expressionEaseIn(NaN, 0, 10, 0, 100) // Returns NaN
```

## Suggested Fix
Same pattern as BUG-182 fix - validate range before division.
